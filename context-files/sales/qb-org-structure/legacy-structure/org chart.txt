POTENTIAL ISSUES FOR REDESIGN
═══════════════════════════════════════════════════════════════════

1. REDUNDANT LOOKUP PATHS
   └─ Projects reach Region via multiple paths:
      • Via Closer → Team → Office → Region
      • Via xx.Region Record ID (direct field)
      • Via Office Record ID
   
2. MIXED FIELD TYPES
   └─ "Closer" exists as BOTH:
      • text field (355)
      • numeric reference (354 - Related Closer)
   
3. MULTIPLE TEAM REFERENCES
   └─ Projects track teams by phase (Survey, Design, Install...)
      └─ But also have a primary "Closer - Team"
      └─ Unclear which team owns the project overall
   
4. CONTACT ROLE ISOLATION
   └─ Contacts have role checkboxes but:
      • Not clearly linked to Teams
      • Multiple projects fields for different roles
      • No clear hierarchy/ownership
   
5. OFFICE/REGION LOOKUP DUPLICATION
   └─ Some fields use Record IDs (809, 810)
   └─ Others use lookups (Office - Name, Region Name)
   └─ Inconsistent reference approach


SUGGESTED QUESTIONS FOR REDESIGN
═══════════════════════════════════════════════════════════════════

1. Primary Project Ownership
   └─ Is a project owned by ONE team/closer, or MULTIPLE teams?

2. Contact-Team Mapping
   └─ Should contacts be linked directly to their team?

3. Project Phase Teams
   └─ Why are teams tracked separately by phase instead of assigned dynamically?

4. Record ID vs Text Fields
   └─ Consolidate to one reference approach?

5. Multi-User Scenarios
   └─ Can an Area Director/District Manager span multiple offices/regions?
